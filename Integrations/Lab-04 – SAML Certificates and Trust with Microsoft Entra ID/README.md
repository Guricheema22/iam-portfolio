# Real-World Understanding

During SAML authentication, the Identity Provider generates and signs the SAML assertion before sending it to the Service Provider application.

The Service Provider validates the digital signature using the trusted signing certificate provided by the Identity Provider.

This trust relationship helps ensure:
- The assertion was generated by a trusted Identity Provider
- The assertion was not modified during transmission
- The authentication response is valid

# Tasks Performed

## 1. Investigated SAML Signing Certificate

Reviewed the SAML Signing Certificate section inside Microsoft Entra ID.

Observed:
- Certificate status
- Thumbprint
- Expiration date
- Federation Metadata URL

The token signing certificate is used by Microsoft Entra ID to digitally sign SAML assertions.

---

## 2. Reviewed Certificate Details

Investigated the certificate thumbprint and expiration date.

### Thumbprint
The thumbprint acts as a unique identifier for the certificate.

### Expiration Date
The expiration date defines how long the certificate remains valid.

Expired certificates can cause SAML authentication failures if not renewed or updated.

---

## 3. Opened Federation Metadata XML

Downloaded and reviewed the Federation Metadata XML file.

The metadata file contains:
- Identity Provider information
- Login URLs
- Entity identifiers
- Signing certificate information
- SAML endpoints

Applications can use this metadata to automatically establish trust with the Identity Provider.

---

## 4. Investigated X509Certificate

Searched the metadata XML file for:
```text
X509Certificate
```

The X509Certificate contains the public signing certificate used by applications to validate SAML assertion signatures.

---

## 5. Reviewed Login URL and Identity Provider Identifier

Reviewed:
- Login URL
- Microsoft Entra Identifier
- Logout URL

The Login URL defines where users authenticate.

The Microsoft Entra Identifier represents the Identity Provider that generates the SAML assertion.

---

# SAML Trust Workflow

1. User attempts to access application  
2. Application redirects user to Microsoft Entra ID  
3. User authenticates successfully  
4. Microsoft Entra ID generates a signed SAML assertion  
5. Service Provider validates the digital signature using the trusted certificate  
6. If signature validation succeeds, application grants access
