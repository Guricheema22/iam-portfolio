# Real-World Understanding

A SAML assertion is a signed authentication message generated by the Identity Provider after successful authentication.

The Service Provider trusts this assertion and uses it to validate:
- User identity
- Authentication status
- Authentication method
- User attributes
- Assertion validity

The assertion also contains conditions and restrictions to ensure it can only be used by the intended application during a valid time window.

---

# Tasks Performed

## 1. Performed SAML Authentication Test

Tested SAML Single Sign-On using the previously configured Enterprise Application in Microsoft Entra ID.

---

## 2. Investigated Subject Information

Reviewed the NameID value inside the SAML assertion to identify the authenticated user.

Example:
```text
user@tenant.onmicrosoft.com
```

The NameID represents the identity of the authenticated user.

---

## 3. Investigated Authentication Details

Reviewed the AuthNStatement section of the assertion.

Observed:
- Authentication timestamp
- Session index
- Authentication method

This section provides information about how and when the user authenticated.

---

## 4. Reviewed Claims and Attributes

Analyzed claims included inside the assertion.

Observed claims such as:
- Display Name
- Email Address
- Given Name
- Surname
- Department
- Tenant ID
- Object Identifier

These claims provide user identity information to the Service Provider application.

---

## 5. Investigated Issuer Information

Reviewed the Issuer value inside the assertion.

Example:
```text
https://sts.windows.net/
```

The Issuer identifies the Identity Provider that generated the SAML assertion.

In this lab, Microsoft Entra ID acted as the Identity Provider.

---

## 6. Investigated Audience Restriction

Reviewed the AudienceRestriction condition.

Example:
```text
https://sptest.iamshowcase.com/metadata
```

AudienceRestriction ensures the assertion can only be used by the intended Service Provider application.

---

## 7. Reviewed Assertion Conditions

Investigated:
- NotBefore
- NotOnOrAfter

These conditions define the validity window of the assertion.

This helps prevent:
- Replay attacks
- Expired authentication reuse
- Unauthorized assertion usage

---

# SAML Assertion Workflow

1. User opened the Service Provider application  
2. Application redirected the user to Microsoft Entra ID  
3. User authenticated successfully  
4. Microsoft Entra ID generated a signed SAML assertion  
5. Assertion included authentication details, claims, and conditions  
6. Assertion was sent to the ACS URL  
7. Service Provider validated the assertion and granted access  

---

# Understanding Assertion Components

## NameID
Identifies the authenticated user.

---

## Issuer
Identifies the Identity Provider that generated the assertion.

---

## AuthNStatement
Contains authentication details such as login time, session information, and authentication method.

---

## AudienceRestriction
Defines which application the assertion is intended for.

---

## Conditions
Define when the assertion is valid and help protect authentication flows.
